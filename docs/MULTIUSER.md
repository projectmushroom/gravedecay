# Multi-user identity and authorization contract

Multi-user mode is opt-in and intended for a small group of trusted
collaborators.  It adds process and credential separation; it is not a
hostile-public-tenancy boundary.

## Identity record

The appliance-owned registry stores records with this shape:

```json
{
  "id": "ts:1234567",
  "login": "dev@example.com",
  "slug": "dev-a1b2c3",
  "role": "developer",
  "enabled": true,
  "projects": [],
  "ports": {"t3": 4810, "term": 4910, "dash": 5010},
  "provider": {"llm": true}
}
```

`id` is the stable Tailscale user ID, prefixed by its authority. `login` is
display metadata and may change. `slug` is generated by the administrator,
validated against `^[a-z][a-z0-9-]{2,30}$`, checked for uniqueness, and is the
only registry value interpolated into Unix names, paths, or unit instances.
Request headers are never used in paths, commands, unit names, ports, or
environment files. Ports are allocated by the control plane and cannot be
selected by a request.

The registry is `$GRAVE_ROOT/config/workspaces.json`, owned by the appliance
administrator and mode 600. Integration credentials live separately in each
workspace's mode-700 private configuration tree.

## Request flow and failures

1. Tailscale Serve terminates HTTPS and supplies its authenticated identity
   headers to the loopback-only gateway.
2. The gateway accepts those headers only on requests arriving through its
   configured trusted proxy path. Direct loopback is reserved for health
   checks and explicit administrator maintenance; it is not an implicit user.
3. The gateway resolves stable user ID first. A login is used only to migrate
   a legacy record after the local Tailscale API confirms the stable ID.
4. The registry selects a fixed upstream for the workspace. Client URLs,
   cookies, forwarded headers, and bodies never select an upstream.
5. Unknown identities receive onboarding (403), disabled identities receive
   disabled (403), and deleted workspaces are unknown. Missing or malformed
   identity is denied (401). A login rename updates display metadata without
   changing the workspace. A shared device resolves to the Tailscale user who
   authenticated the request, not an email or device name.

Local `/healthz` checks require no identity and disclose no user state. All
other local access must use the administrator maintenance mechanism described
by the gateway; blindly trusting headerless loopback would let any workspace
process impersonate the administrator.

## Roles and endpoint classes

| Capability | admin | developer |
|---|---:|---:|
| Own T3, terminal, projects, settings, GitHub/Linear | yes | yes |
| Pair or revoke own integrations | yes | yes |
| Appliance status (sanitized) | yes | yes |
| Users, grants, provider policy, global settings | yes | no |
| Upgrade, reboot, modes, units, global logs/files | yes | no |
| Skip-permissions/yolo agents | owner policy | no |

Dashboard state, T3 HTTP/WebSocket, terminal, pairing, files, settings,
provider sessions, and integration caches are developer-scoped. Appliance
actions, unrestricted file management, service control, pairing
administration, upgrades, and user management are admin-only. The old shared
terminal and global file manager are removed from developer routes.

## Trust boundaries

- Tailscale ACLs/grants decide who can reach the appliance.
- Tailscale Serve authenticates the remote principal.
- The gateway maps that principal to role and a fixed workspace.
- Unix ownership prevents ordinary cross-workspace file, process-state, home,
  credential, tmux-socket, and log access.
- Project grants limit what the dashboard and T3 expose; repository-provider
  authorization remains the final control for clone and push.

Unix users receive no `sudo`, wheel, docker, journal, or appliance-owner group
membership. The owner account remains effectively root-equivalent. v1 assumes
trusted collaborators: kernel exploits, denial of service, and extraction of
a directly delivered shared provider credential are outside its hard boundary.

## Audit contract

Append JSON Lines under `$GRAVE_ROOT/logs/audit.jsonl`, mode 600. Events include
timestamp, event name, actor stable ID/workspace, target workspace/project, and
result. Required events are provisioning/removal,
enable/disable and role/grant/provider-policy changes, integration
login/logout/failure, session creation, denied access, and every appliance
administrative action. Tokens, authorization headers, cookies, pairing URLs,
command output, secret values, and request bodies are never recorded.

## Migration

Single-user mode remains the default. Enabling multi-user creates an `admin`
owner record using the locally confirmed Tailscale stable ID, adopts existing
T3 state and repositories without moving them until backup succeeds, then
switches Serve to the gateway. Failure before the switch leaves single-user
services intact. Rollback restores the previous Serve configuration and
units; workspace data is retained. Ambiguous identity, duplicate IDs/slugs or
ports, unsafe ownership, and an unconfirmed owner ID stop migration.

## Operations

Migration is explicit: `grave multiuser enable <stable-id> <owner-login>
owner --profile <profile>`. It takes a normal backup plus a migration snapshot,
creates the admin workspace without starting it, copies owner state and adopts
existing Git remotes, then sets `MULTI_USER=1` and re-runs the ritual. Serve
changes only during successful re-raise. Failure restores single-user config
and retains the copied workspace for inspection.

Administrators use `grave users`, `grave projects`, `grave integrations`, and
`grave provider` for lifecycle, grants, onboarding and entitlement. Run
`grave users status` and `grave doctor` after changes. Developers are confined
to their routed dashboard/T3/terminal and private HOME; they cannot manage
users, grants, provider policy, modes, upgrades, reboot, global files/units,
or unrestricted logs. Disablement stops all workspace units immediately;
removal requires prior disablement and archives rather than deletes its home.

`grave backup` includes registry-backed workspace state and dirty work but
excludes credentials by default. `--include-secrets` is an explicit sensitive
choice. `grave restore <timestamp> workspaces`, followed by re-raise and doctor,
reproduces grants, service config, checkouts and state; omitted credentials
require reauthentication.
